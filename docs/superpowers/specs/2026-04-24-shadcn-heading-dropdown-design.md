# Design: Refactor Heading Dropdown Menu with shadcn Components

**Date:** 2026-04-24
**Scope:** `packages/notra-editor`
**Status:** Draft

## Goal

Migrate `heading-dropdown-menu` from the legacy stack (custom `Button` + hand-rolled `ui-primitive/dropdown-menu` + custom SVG icons) to the shadcn stack (shadcn `Button` + shadcn `DropdownMenu` backed by `radix-ui` + `lucide-react` icons). This continues the incremental shadcn migration started with the undo/redo buttons.

## Scope

In-scope:

- `src/components/heading-dropdown-menu/` — all three files migrated
- New `src/components/ui/dropdown-menu.tsx` — generated by shadcn CLI
- Deletion of heading-specific icon files that become unused

Out of scope (kept for future passes):

- `src/components/list-dropdown-menu/` — still uses the legacy stack
- `src/components/ui-primitive/dropdown-menu.tsx` — still consumed by list-dropdown
- `src/components/button/button.tsx` (legacy Button) — still consumed by list-dropdown, mark-button, etc.
- `src/icons/chevron-down-icon.tsx`, `src/icons/check-icon.tsx` — still referenced by list-dropdown and link-popover
- CSS theme variables (`--tt-*`) — untouched

## Constraints

- Public API of `HeadingDropdownMenu` must stay backward compatible at the call-site level (`editor` and `levels` props unchanged).
- No new runtime dependency is added: `radix-ui` meta package (v1.4.3) already re-exports `DropdownMenu`; the shadcn dropdown component imports from it as `import { DropdownMenu as DropdownMenuPrimitive } from 'radix-ui'`, matching the project's existing pattern (see `src/components/ui/button.tsx` using `Slot` from `radix-ui`).
- `lucide-react` is already a dependency.
- All Tailwind classes keep the `nt:` prefix.
- The `use-heading.ts` hook's editor-state logic (selection listeners, `canToggleHeading`, `handleToggle`) stays unchanged. Only the icon map is swapped.

### Renamed + rewritten: `heading-button.tsx` → `heading-menu-item.tsx`

The old file wrapped a legacy `Button` with a `role="menuitem"` string, which does not integrate with Radix's focus/keyboard manager. The new file returns a `DropdownMenuItem` directly.

```tsx
import { Check } from 'lucide-react';
import { forwardRef } from 'react';

import { useHeading, type HeadingLevel } from './use-heading';
import { DropdownMenuItem } from '../ui/dropdown-menu';

import type { Editor } from '@tiptap/core';

export interface HeadingMenuItemProps {
  editor: Editor | null;
  level: HeadingLevel;
}

export const HeadingMenuItem = forwardRef<HTMLDivElement, HeadingMenuItemProps>(
  ({ editor, level }, ref) => {
    const { isActive, canToggle, handleToggle, label, Icon } = useHeading({ editor, level });

    return (
      <DropdownMenuItem
        ref={ref}
        aria-label={label}
        data-active-state={isActive ? 'on' : 'off'}
        disabled={!canToggle}
        onSelect={handleToggle}
      >
        <Icon />
        <span>{label}</span>
        {isActive && <Check className="nt:ml-auto" />}
      </DropdownMenuItem>
    );
  }
);

HeadingMenuItem.displayName = 'HeadingMenuItem';
```

Notable changes vs. old `HeadingButton`:

- `forwardRef` target is `HTMLDivElement` (Radix renders `DropdownMenuItem` as a div with `role="menuitem"`).
- Event handler is Radix's `onSelect` (fires on click + Enter/Space) wired directly to `handleToggle`, instead of `onClick`.
- Dropped hand-written `role="menuitem"`, `tabIndex`, `type="button"`, `aria-pressed` — all handled by Radix.
- Dropped `tiptap-button-icon` className on the icons — shadcn `DropdownMenuItem` already has `[&_svg]:size-4` in its recipe. `Check` is pushed to the right via `nt:ml-auto`.
- `HeadingMenuItem` is an internal component (not exported from `src/index.ts`), so the prop type change is invisible to consumers.

### Rewritten: `heading-dropdown-menu.tsx`

```tsx
import { ChevronDown } from 'lucide-react';
import { forwardRef } from 'react';

import { HeadingMenuItem } from './heading-menu-item';
import { getHeadingTriggerIcon, useActiveHeadingLevel, type HeadingLevel } from './use-heading';
import { Button } from '../ui/button';
import {
  DropdownMenu,
  DropdownMenuTrigger,
  DropdownMenuContent,
  DropdownMenuGroup
} from '../ui/dropdown-menu';

import type { Editor } from '@tiptap/core';
import type { ComponentProps } from 'react';

export interface HeadingDropdownMenuProps extends Omit<ComponentProps<typeof Button>, 'type'> {
  editor: Editor | null;
  levels?: HeadingLevel[];
}

export const HeadingDropdownMenu = forwardRef<HTMLButtonElement, HeadingDropdownMenuProps>(
  ({ editor, levels = [1, 2, 3, 4], ...buttonProps }, ref) => {
    const activeLevel = useActiveHeadingLevel(editor, levels);
    const TriggerIcon = getHeadingTriggerIcon(activeLevel);

    return (
      <DropdownMenu>
        <DropdownMenuTrigger asChild>
          <Button
            ref={ref}
            aria-label="Heading"
            data-active-state={activeLevel !== null ? 'on' : 'off'}
            size="icon"
            tabIndex={-1}
            type="button"
            variant="ghost"
            {...buttonProps}
          >
            <TriggerIcon />
            <ChevronDown className="nt:size-3" />
          </Button>
        </DropdownMenuTrigger>
        <DropdownMenuContent align="start">
          <DropdownMenuGroup>
            {levels.map((level) => (
              <HeadingMenuItem key={level} editor={editor} level={level} />
            ))}
          </DropdownMenuGroup>
        </DropdownMenuContent>
      </DropdownMenu>
    );
  }
);

HeadingDropdownMenu.displayName = 'HeadingDropdownMenu';
```

Notable changes:

- Trigger is now a Radix `DropdownMenuTrigger asChild` wrapping a shadcn `Button`; `ref` still forwards to the underlying `<button>`.
- `size="icon"` + `variant="ghost"` matches the undo/redo buttons.
- The hand-written `tiptap-button-icon` / `tiptap-button-dropdown-arrows` classes are removed; sizing is handled by the Button recipe and explicit `nt:size-3` on the chevron.
- Prop type now extends `ComponentProps<typeof Button>` instead of `React.ButtonHTMLAttributes<HTMLButtonElement>`. This is a superset in practice (adds `variant`, `size`, `asChild`) and does not break existing call sites.

## Public API Impact

- `HeadingDropdownMenu` is still exported from `src/index.ts` with the same name; the `editor` and `levels` props keep their meaning.
- `HeadingDropdownMenuProps` extends `ComponentProps<typeof shadcnButton>` — a superset of the previous `ButtonHTMLAttributes`, so existing consumer code compiles without change. New opt-in props (`variant`, `size`) become available.
- Visual style shifts from legacy CSS classes to shadcn defaults (`ghost` + `icon`), matching the undo/redo buttons already shipped. This is the intended unification, not a regression.
- The internal `HeadingButton` (never re-exported) is replaced by `HeadingMenuItem`, also internal — invisible to consumers.

## Testing / Verification

- `pnpm --filter notra-editor test` — existing unit coverage continues to pass; hook logic is unchanged.
- `pnpm --filter notra-editor build` — the tsup + PostCSS pipeline produces `dist/styles/globals.css` containing the new `[data-slot=dropdown-menu-content]` / `[data-slot=dropdown-menu-item]` rules.

Manual checklist (in the demo app):

1. Click trigger — menu opens, positioned `align="start"` relative to trigger.
2. Click H1/H2/H3/H4 — editor converts current block to that heading level.
3. Click the already-active level — block converts back to paragraph.
4. Place cursor in an H2 — trigger icon shows `Heading2`, and the H2 menu item shows a right-aligned `Check`.
5. Keyboard: Tab to trigger, Enter opens, Arrow keys move focus, Enter selects, Esc closes.
6. Click outside — menu closes.
7. Set editor read-only (`editor.setEditable(false)`) — menu items render disabled; trigger still opens (preserves current behavior).

## Open Question (decide during implementation)

The trigger uses `size="icon"` (fixed 8×8) holding both the heading icon and the chevron. If this feels cramped in practice, fall back to `size="sm"` with `className="nt:w-auto nt:px-1.5"` so the trigger grows to fit. Spec specifies `size="icon"` as the default; reassess on first visual check.
